This is a summary of libraries needed to implement the
requirements.  

libraries which need specific versions for compatibility:
  - apache-beam
  - TFX
  - TensorFlow Data Validation
  - TensorFlow Transform
  - tfx_bsl

need to use a conda virtual environment due to python 3.10 requirement
for TFX.
pip -q install pyarrow==10.0.1
pip -q install apache-beam==2.59.0
pip -q install tensorflow==2.16.1
pip -q install tfx==1.16.0
#pip -q install tensorflow-transform==1.16.0
tfx installs many dependencies listed from the github README or pypi 

TFX is not available for all architectures.  e.g. apple arm m4 pro.

TFX is compatible with a restricted range of python and tensorflow.
see https://pypi.org/project/tfx/
use python 3.9 or 3.10

Note that the compatibilities of versions are such that
tensorflow 2.16 is used.
That makes cannot use tensorflow ranking because 
TFR 0.6.0 is needed for tf 2.16 and then the
versions of keras would conflict.
So any ranking metrics and losses used here are implemented.

Data distributed data pre-processing and transformation:
- data ingestion:
  - apache beam components in TFX pipeline are highly distibutable
  and there are options and many examples on how to make custom
  components with dsl annotatings as needed
  - custom component is needed to join the ratings records
    with user and movie information.
- transformations
  - normalizing rating to a relevance value between 0 and 1
  - transforming timestamp into features that might be used in the model.
  - options for library to use:
    - tf.data is for in-memory size data processing so not an option here.
    - considered using keras layers
      pros:
        - less code, simpler to maintain
      cons:
       - by performing the transformation to features in the model, we are
         repeating work, especially during the compute intensive hyper-parameter
         tuning phase.
       - features, that is, tranformed data, are not saved in a feature store
       - cannot perform stats on features
    - decided to use tft and tfx
      pros:
        - transform to features once and that is input to the hyper-parameter tuning
        - transformed data is persisted and usable in calculating statistics
      cons:
        - ramping-up to using it is larger

pipeline:
  - decided on TFX 
    - scalability, works on linux in all environments, is a mature platform with
      alot of documentation, examples, community support
    - the pipeline can be compiled to create a yaml intermediate file
      for running with airflow or kfp, etc.  cloud specific changes
      for Vertex AI, etc. should be straight forward.
  - the components needed in the pipeline to implement the requirements:
    - MovieLensExampleGen: a custom ingest component that joins ratings with 
      user and movie data
    - Resolver for resolving artifacts in the MLMD datastore
    - StatisticsGen for calculating stats on the transformed data
    - SchemaGen for extracting a schema from the statistics
    - Importer for inporting a schema when needed
    - ExampleValidator for anomaly and missing value detection
    - ExampleDiff for looking for data skew compared to previous data
    - Tranform to perform data transformations
    - Tuner to find hyper-parameters for best performing model
    - Trainer to train a model using the best hyper-parameters
    - Evaluator for calculating model eval stats and a quality 
      validation w.r.t. a baseline model
    - optionally: InfraValidator to validate that the model 
      is compatible with the serving infrastructure.
    - Pusher for pushing a blessed model to the serving_model directory
    - BulkInferrerBeam: a custom component to handle specifics for saved_model

monitoring:
  - any metrics monitored outside of the pipeline, for a model actively
    serving inference.

listwise evaluation metrics:
  - comparing current model to previous:
    - normalized hit rate (NHR)
  - comparing each model in the series of Retrieval, Ranking, and Re-Ranking
    for improvments in ability to provide relevant ranked recommendations
    to a user:
    - NHR, NDCG@K, recall_tail@K, and intra-list diversity

Misc:
  - ability to write data in formats: tfrecord, parquet, and array_record
    for downstream components

Late item of interest regarding data preparation before
the pipeline:
   - movie ids are rewritten from src/main/resources/ml-1m-orig/
     to src/main/resources/ml-1m/ by script
     src/test/python/movie_lens_tfx/RenumberIds.py
     - the movie ids are reassigned numbers that start at n_users + 1
       and continue consecutively to n_user + n_movies + 1.
     - the reason that the re-numbering is needed is because a
       downstrem component, the Ranker, needs the movie ids to follow
       the user ids in numerical order as they are both used as
       indices (offset by 1) for accessing a matrix of concatenated 
       user_embeddings followed by movie_embeddings.
